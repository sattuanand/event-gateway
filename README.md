# event-gateway

Ingests financial events (CREDIT/DEBIT), durably stores them, and forwards each one to
`account-service` exactly once — with idempotent resubmits and a circuit breaker guarding
the downstream call.

## Architecture

Two independent Spring Boot services, each owning its own Postgres database, talking over
plain HTTP:

```
client → [event-gateway :8080] → [account-service :8081]
              │                          │
          eventledger DB             accountledger DB
```

- **`event-gateway`** (this repo) is the public entry point. It validates and durably stores
  every incoming event *before* calling downstream, keyed by client-supplied `eventId`. A
  resubmit of the same `eventId` returns the original stored result instead of re-processing —
  that's what makes the whole pipeline safe to retry at any layer, including from real client
  retries, not just internally.
- **`account-service`** owns account balances and the transaction ledger. It's the source of
  truth for "did this money move" — `event-gateway` never mutates a balance itself, it only
  ever asks account-service to.
- The **hop between them is also idempotent**, by the same `eventId`: if `event-gateway`
  retries a call (timeout, 5xx, or a resubmitted client request), account-service returns the
  transaction it already applied rather than double-applying it. Idempotency has to hold at
  both hops for the "exactly once" guarantee to mean anything — a client-facing idempotency key
  that isn't also carried on the downstream call just moves the double-apply problem one level
  down.
- A **W3C `traceparent`** header is generated by event-gateway and forwarded on the outbound
  call, so one trace ID ties a single client request to log lines in both services (see
  `X-Trace-Id` on every response, and the `traceId` field in every JSON log line).
- See account-service's own README for its API and internals, and the "Resiliency pattern"
  section below for how event-gateway handles account-service being slow, unreachable, or
  actively failing.

## System requirements

- Docker Desktop (or Docker Engine + Compose v2) — everything runs in containers, no local
  JDK required to run the stack.
- A clone of [`account-service`](../account-service) alongside this repo — event-gateway
  depends on it at runtime and does not stand it up itself.
- Only needed if you plan to build/run outside Docker: JDK 21 and Gradle (wrapper included).

## Starting the system

### Single command (`docker-compose.full.yml`)

Builds and runs both services plus both Postgres instances in one project — no external
network to create by hand. Requires the `account-service` checkout alongside this repo (see
System requirements above), and must be run from this directory since the file's
`account-service` build context is `../account-service`.

```bash
docker compose -f docker-compose.full.yml up --build
```

### Two independent stacks (`docker-compose.yml`)

`account-service` and `event-gateway` can also run as independent Compose stacks that
communicate over a shared external Docker network — useful if you want to start, stop, or
rebuild either service without touching the other. Create the network once, start
`account-service` first, then `event-gateway`.

```bash
# 1. One-time: create the shared network both stacks join
docker network create event-ledger-net

# 2. Start account-service (its own Postgres + the service)
cd ../account-service
docker compose up -d --build

# 3. Start event-gateway (its own Postgres + the gateway)
cd ../event-gateway
docker compose up -d --build
```

Ports exposed on localhost:

| Service              | Port |
|----------------------|------|
| event-gateway         | 8080 |
| account-service       | 8081 |
| event-gateway postgres | 5432 |
| account-service postgres | 5433 |

Check both are healthy:

```bash
curl.exe -s localhost:8080/actuator/health
curl.exe -s localhost:8081/health
```

## Running the tests

```bash
./gradlew test
```

The suite (`src/test/java`) is almost entirely integration-level: a real Postgres via
Testcontainers (so the idempotency guarantee is verified against the actual unique-constraint
behavior it relies on, not an H2 approximation of it) and WireMock standing in for
account-service, which lets tests force the downstream into specific failure modes — timeout,
5xx, connection refused — on demand. No Docker Compose stack needs to be running first;
Testcontainers starts and tears down its own Postgres container per test run. Docker itself does
need to be available, since Testcontainers launches a real container.

One test, `FullStackIntegrationTest`, is different: it doesn't use the WireMock stub at all. It
builds the real Account Service image straight from `../account-service`'s Dockerfile (see
"System requirements" below for the sibling-checkout prerequisite), runs it with its `standalone`
profile (embedded H2, no second Postgres needed), and drives a real request through the Gateway
into it — the automated equivalent of the manual curl walkthrough further down this README. It
runs as part of `./gradlew test` like everything else, but since it does a real Docker image
build (a full Gradle build inside the container) rather than just starting a pre-built image, it
is the slowest test in the suite — several minutes on a cold Docker build cache, seconds after
that cache is warm. To run just this one while iterating:

```bash
./gradlew test --tests "com.eventledger.gateway.FullStackIntegrationTest"
```

## API

| Method | Path             | Purpose                              |
|--------|------------------|---------------------------------------|
| POST   | `/events`        | Submit an event (idempotent by `eventId`) |
| GET    | `/events/{id}`   | Fetch a stored event                  |
| GET    | `/events?account={accountId}` | List events for an account, chronological |
| GET    | `/accounts/{accountId}` | Current balance — proxies to account-service; `503` if it's unreachable, `404` if the account genuinely doesn't exist |

## Sample curl

Write the payload to a file first — passing multi-line JSON inline to `curl.exe` on
Windows/PowerShell is unreliable (embedded quotes get mangled by native-argument passing).

```powershell
@'
{
  "eventId": "evt-001",
  "accountId": "acct-977",
  "type": "DEBIT",
  "amount": 100.00,
  "currency": "USD",
  "eventTimestamp": "2026-05-15T08:30:11Z",
  "metadata": {"source": "mainframe-batch"}
}
'@ | Set-Content evt.json

curl.exe -X POST localhost:8080/events -H "Content-Type: application/json" -d "@evt.json"
```

Resubmitting the same file returns the original stored event (HTTP 200, unchanged
`receivedAt`) instead of re-applying it downstream — that's the idempotency guarantee, not a
bug.

Fetch it back:

```bash
curl.exe -s localhost:8080/events/evt-001
curl.exe -s "localhost:8080/events?account=acct-977"
```

## Running standalone, without Docker

Each service can also run as a bare jar with its own embedded, in-memory H2 database instead
of Postgres — two independent processes, no Docker, no shared state, nothing to install beyond
a JDK. Data does not survive a restart.

```bash
# account-service
cd account-service
./gradlew bootJar
java -jar build/libs/account-service.jar --spring.profiles.active=standalone

# event-gateway (separate terminal)
cd event-gateway
./gradlew bootJar
java -jar build/libs/event-gateway.jar --spring.profiles.active=standalone
```

This uses the `standalone` Spring profile (`application-standalone.yml` in each repo), which
points `spring.datasource.url` at `jdbc:h2:mem:...` instead of Postgres. Everything else —
ports, `ACCOUNT_SERVICE_URL`, Flyway migrations — is unchanged; the same SQL migrations that
run against Postgres in Docker run against H2 here. Verified end-to-end: submit an event via
`POST :8080/events`, and `GET :8081/accounts/{accountId}` on the standalone account-service
reflects the correct balance, with each process owning a completely separate database.

## Configuration

Set via environment variables on the `gateway` service in `docker-compose.yml`:

| Variable              | Default (in-container)         | Purpose                     |
|-----------------------|----------------------------------|------------------------------|
| `DB_URL`              | `jdbc:postgresql://postgres:5432/eventledger` | Gateway's own Postgres |
| `DB_USERNAME`         | `eventledger`                    | —                            |
| `DB_PASSWORD`         | `eventledger`                    | —                            |
| `ACCOUNT_SERVICE_URL` | `http://account-service:8081`    | Base URL for the downstream call — must resolve on `event-ledger-net`, not `localhost` |
| `OUTBOX_SWEEPER_ENABLED` | `true` | Set `false` to disable the outbox sweeper entirely (see below) — the bean isn't created at all in that case |
| `OUTBOX_SWEEPER_INTERVAL_MS` | `300000` (5 min) | How often the sweeper runs |
| `OUTBOX_SWEEPER_STALE_AFTER` | `PT2M` | A `PENDING` row untouched this long is assumed orphaned by a crash and reaped to `FAILED` |
| `OUTBOX_SWEEPER_BATCH_SIZE` | `50` | Max `FAILED` events redriven per sweep |
| `OUTBOX_SWEEPER_MAX_ATTEMPTS` | `10` | A row that has failed this many redrive attempts is left alone (still resubmittable manually) rather than retried forever |

## Automatic recovery: the outbox sweeper

An event that ends up `FAILED` — because account-service was down when it was submitted — used to
require the *client* to resubmit the same `eventId` for the balance change to ever actually happen.
`OutboxSweeper` (`service/OutboxSweeper.java`) closes that gap: a `@Scheduled` job that periodically
redrives `FAILED` events (and reaps any `PENDING` row orphaned by a crash mid-call) through the
exact same idempotent, Retry+CircuitBreaker-wrapped call a client resubmit uses — no new
infrastructure, no new failure mode, same CAS guard against double-applying.

It's fully configurable via the `OUTBOX_SWEEPER_*` variables above, and can be disabled outright
with `OUTBOX_SWEEPER_ENABLED=false`. A chronically-failing event stops being auto-retried once it
exceeds `OUTBOX_SWEEPER_MAX_ATTEMPTS` redrive attempts (tracked in `events.redrive_count`) — it's
still resubmittable directly by a client at any time, the sweeper just stops looping on it forever.
See [WIKI.md §6](WIKI.md#6-async-recovery-outbox-sweeper-built-and-a-possible-kafka-future-not-built)
for the full design, including why the stale-`PENDING` reap is a separate step from the `FAILED`
redrive.

## Resiliency pattern

The outbound call to account-service (`AccountServiceClient.applyTransaction`) is wrapped in
**Resilience4j retry + circuit breaker**, in that order — Resilience4j's documented default
aspect order is `Retry(CircuitBreaker(call))`, i.e. retry is the outermost layer:

- **Circuit breaker** trips on a count-based sliding window (10 calls in production,
  configurable per environment) once the failure rate crosses 50%, then opens for a wait
  duration before probing again with a limited number of half-open calls. While open, calls
  fail immediately with `CallNotPermittedException` instead of hitting a downstream that's
  already known to be struggling — the whole point of the pattern: stop making a bad situation
  worse by piling more load onto a service that's failing.
- **Retry** wraps the breaker with exponential backoff and jitter (3 attempts, doubling delay,
  ±50% randomization) so that a transient blip — one dropped connection, one slow GC pause —
  self-heals without becoming a user-visible error, while jitter keeps many concurrent retries
  from re-stampeding the recovering service in lockstep.
- **Both explicitly exclude 4xx responses.** A `HttpClientErrorException` (account-service
  rejecting the request — bad currency, insufficient funds) means account-service is healthy
  and the *request* was wrong. Retrying it would just get the same rejection three times
  instead of once; counting it against the breaker would trip the breaker on bad client input,
  not on an actual outage. Only 5xx, connection failures, and read timeouts count as failures
  for both retry and the breaker.
- **A short, explicit read timeout** (`RestClientConfig`) is what makes any of this meaningful.
  A circuit breaker cannot protect you from a downstream that's slow rather than dead — without
  a timeout, "slow" just means every gateway request thread parks on a socket, and the breaker
  never sees a failure to count because no call ever actually fails, it just never returns.
- If retries are exhausted or the breaker rejects the call outright, the event is durably
  persisted as `FAILED` rather than lost — it already exists in `event-gateway`'s own database
  from before the downstream call was attempted, so a client's `202`-then-poll or resubmit flow
  can recover it without event-gateway needing to buffer anything in memory.

This was chosen over a message-queue-based retry (e.g. publish to a queue and let a consumer
retry independently) because the requirement is synchronous, idempotent request/response
between two services that are both up the overwhelming majority of the time — a breaker+retry
pair handles that with far less operational surface area (no broker to run, no dead-letter
queue to monitor) than an async pipeline would, at the cost of not surviving an outage longer
than the retry budget. Given `event-gateway` durably persists the event before attempting the
downstream call either way, nothing is lost even in that longer-outage case — it just requires
a resubmit or a background reconciliation job to pick up `FAILED` events, rather than that
recovery happening automatically.
